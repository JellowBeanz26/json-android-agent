## 🎯 המטרה הסופית (ה-North Star)
אפליקציית אנדרואיד **עצמאית** — רצה על הטלפון עצמו, **בלי תלות ב-PC** — שמבצעת כל משימה שהמשתמש מבקש, ב**טקסט** וב**קול**, על ידי שליטה במסך דרך `AccessibilityService`, עם "מוח" שרץ **בענן** (Gemini). שימוש **אישי** (התקנה עצמית / sideload).

## 🧭 אסטרטגיה: שתי פאזות
פרוטוטייפ מהיר ב-Python (מוכיח את הלוגיקה) ← ואז בנייה מחדש כאפליקציית Kotlin (המוצר האמיתי).
**למה:** הלוגיקה הקשה (הלולאה, הפרומפטים, הכלים) מתפתחת פי-כמה מהר ב-Python; רק כשמוכח *מה* עובד — בונים ביעד הסופי. זה מונע היתקעות, ומהפך את בניית ה-Kotlin מ"ירייה באפלה" ל"תרגום של משהו מוכח".

✅ **A1–A2 הושלמו — Json עובד!** רץ מקצה-לקצה: קורא מסך → Gemini מחליט → מבצע פעולה → מדווח done. משימות פשוטות (פתיחת אפליקציה) עובדות נקי.
- **מוח:** Gemini דרך **Vertex AI** (על הקרדיטים שלך ב-CoWork) — כי הארגון חוסם את ה-Gemini API החינמי. `.env`: `USE_VERTEX=true`, מודל `gemini-2.5-flash`.
- **הצעד הבא: A3 — כיוונון אמינות** על משימות רב-שלביות (המחשבון עדיין מגמגם).

## ⚠️ סיכונים עיקריים ואיך מתמודדים
- **שרשור טעויות** (95%^15 ≈ 46%) → verify אחרי כל צעד, retry, תקרת צעדים.
- **עקומת למידה של Kotlin** (פאזה B) → הפרוטוטייפ המוכח מקל דרמטית.
- **עץ נגישות ריק** באפליקציות מסוימות → גיבוי ראייה.
- **פעולות הרסניות** (שליחה/מחיקה) → אישור דרך `ask_user`.
